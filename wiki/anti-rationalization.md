# Anti-Rationalization

**TL;DR** LLM 极其擅长合理化——生成听起来完全合理的文字解释为什么这次不需要规格/测试/评审。反合理化表格是预判 agent 自我辩护、提前写下反驳的机制。

## What it means

Agent 知道规则。但它会生成一段听起来很合理的文字，解释为什么这个特定任务不需要规格，或者为什么这个特定改动可以不经评审就合并。这不是 bug——这是 LLM 的生成特性：给定一个捷径的上下文，它能产出极具说服力的自我辩护。

反合理化表格就是在 agent 还没来得及说话之前，就把它的谎言堵回去：

| 常见借口 | 反驳 |
|---|---|
| "太简单，不需要规格" | 验收标准仍然适用。五行可以。零行不行。 |
| "稍后再写测试" | "稍后"是最危险的关键词。没有稍后。先写失败的测试。 |
| "测试通过了，发布吧" | 通过是证据，不是证明。检查运行时了吗？有人类读过 diff 吗？ |

 ([[agent-skills-osmani]])

## 为什么这不是 redundancy

你可能会想：规则已经写了"要写测试"，再写一张表说"如果你想说不需要测试，你是错的"有什么用？

区别在于**对抗性**。规则是声明式的（"做 X"），反合理化是对抗性的（"当你试图解释为什么不做 X 时，你的理由是错的"）。后者在 agent 已经在生成合理化文字时拦截它——规则已经被绕过了，反合理化在绕过点上堵住。

这和 [[pass-state-gating]] 的逻辑一致：规则说"测试通过才能标记 done"，但 [[overconfidence-bias|agent 的过度自信]]会让它说"测试看起来会通过"。Pass-state gating 在结构上阻止自评，反合理化表格在文本上阻止自辩。两道防线，不同层。

## 对人类团队同样有效

大多数工程退化不是因为有人选择做坏工作，而是因为人们接受了听起来合理的理由跳过不想做的部分。"我们上线后再修测试""这个改动太小不需要设计文档""没事我们有监控"——每个都是真话，每个都是陷阱。

一个会写下自己反合理化清单的团队，会少很多这样的借口。把清单放进 AGENTS.md 或工程 wiki，省掉争论，抓住疲惫周五下午的捷径。([[agent-skills-osmani]])

## 和 wiki 其他机制的关系

| 机制 | 防线位置 | 对抗什么 |
|---|---|---|
| [[three-layer-termination]] | 结果层 | agent 跳过验证就宣布完成 |
| [[pass-state-gating]] | 结构层 | agent 自己给自己打分 |
| 反合理化表格 | 文本/决策层 | agent 生成合理化绕过规则 |

三者互补：反合理化在决策时拦截，pass-state gating 在结构上阻止，三层验证在结果上抓捕。
